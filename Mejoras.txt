Comparativa real de carga
Con polling cada 3s:

1 usuario conectado = 20 peticiones HTTP por minuto
100 usuarios = 2.000 peticiones por minuto
Cada petición abre conexión, autentica, consulta BD, cierra conexión
La mayoría devuelven exactamente los mismos datos que la vez anterior

Con WebSocket:

1 conexión persistente por usuario, se abre una vez
0 peticiones si no hay mensajes nuevos
El servidor solo habla cuando hay algo que decir
100 usuarios = 100 conexiones abiertas, silenciosas hasta que alguien escribe


Actualmente tiene el canal inutilizado
Lo más probable es que el canal se implementara con la intención de usarlo, pero el polling se dejó como solución provisional y nunca se reemplazó. Es un patrón muy común: el WebSocket está correctamente configurado en el backend (el evento, el canal, la autorización), pero el frontend nunca se suscribe a él.
La solución es sustituir el setInterval del useEffect de mensajes por la suscripción con Echo que te mostré arriba. El polling de la lista de conversaciones sí tiene más sentido mantenerlo (a un intervalo mayor), porque ahí no tienes un canal WebSocket configurado para notificar conversaciones nuevas.

SOLICITUD (requestPurchase):
Comprador pulsa "Comprar"
  │
  ├─► ¿Producto visible y disponible? ── No ──► 422
  ├─► ¿Es su propio producto?         ── Sí ──► 422
  ├─► ¿Tiene saldo suficiente?        ── No ──► 422
  │
  └─► PurchaseRepository::createOrder() → Order (status: pendiente, escrow: true)
        └─► Product::available = 'reservado'
              └─► Devuelve la Order al controlador

CONFIRMACIÓN (confirmPurchase):
Vendedor pulsa "Confirmar"
  │
  ├─► ¿La orden es suya y está pendiente? ── No ──► 404
  │
  ├─► Order::status = 'completado', escrow_active = false
  ├─► Product::available = 'vendido'
  │
  └─► event(new PurchaseConfirmed($order))
        │
        └─► DeductBuyerWallet::handle()
              ├─► WalletRepository::subtractBalance(comprador, precio, 'pago_pedido')
              │     ├─► UPDATE wallets SET balance = balance - precio
              │     └─► INSERT transactions (tipo: pago_pedido, order_id)
              │
              └─► WalletRepository::addBalance(vendedor, precio, 'cobro_pedido')
                    ├─► UPDATE wallets SET balance = balance + precio
                    └─► INSERT transactions (tipo: cobro_pedido, order_id)

RECHAZO (rejectPurchase):
Vendedor pulsa "Rechazar"
  │
  ├─► Order::status = 'cancelado', escrow_active = false
  └─► Product::available = 'disponible'  (ningún movimiento de dinero)
